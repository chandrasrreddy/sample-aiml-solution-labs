# Requirements Document

## Introduction

This feature adds YAML-based configuration file support to `bedrock_pricing.py`, allowing users to set persistent defaults for reports, agent parameters, AgentCore infrastructure, business value assumptions, capacity planning, pricing data caching, and behavioral preferences. Configuration is opt-in — the script works identically without any config file present. A layered precedence model (function parameter > env var > project config > user config > hardcoded defaults) ensures flexibility across environments.

## Glossary

- **Config_Loader**: The module responsible for discovering, reading, parsing, and merging YAML configuration files into a unified settings dictionary.
- **User_Config**: The user-level configuration file located at `~/.bedrock_skills/config.yaml`.
- **Project_Config**: The project-level configuration file located at `./.bedrock_skills.yaml` in the current working directory.
- **Merged_Config**: The final configuration dictionary produced by merging User_Config and Project_Config according to precedence rules.
- **Config_Template**: A commented YAML file generated by `--init-config` that documents all available settings with their defaults.
- **Precedence_Chain**: The resolution order for any setting: function parameter > environment variable > Project_Config > User_Config > hardcoded default.
- **Bedrock_Pricing_Script**: The main `bedrock_pricing.py` script (~4293 lines) that performs pricing queries, cost calculations, and capacity planning.
- **Pricing_Cache**: The local JSON files in `~/bedrock_cache/` containing pricing and quota data fetched from the AWS Pricing API. Not to be confused with prompt caching (a Bedrock inference feature that reduces token costs).

## Requirements

### Requirement 1: Load and Parse Configuration Files

**User Story:** As a user, I want the script to automatically load my YAML configuration files at startup, so that my preferred defaults are applied without passing them as arguments every time.

#### Acceptance Criteria

1. WHEN the Bedrock_Pricing_Script is invoked, THE Config_Loader SHALL search for User_Config at `~/.bedrock_skills/config.yaml`
2. WHEN the Bedrock_Pricing_Script is invoked, THE Config_Loader SHALL search for Project_Config at `./.bedrock_skills.yaml` in the current working directory
3. WHEN both User_Config and Project_Config exist, THE Config_Loader SHALL perform a recursive deep merge where Project_Config values take precedence over User_Config values at every nesting level, such that keys present in User_Config but absent in Project_Config are preserved in the Merged_Config
4. WHEN neither User_Config nor Project_Config exists, THE Config_Loader SHALL return an empty configuration and the Bedrock_Pricing_Script SHALL use hardcoded defaults for all settings
5. WHEN only one configuration file exists, THE Config_Loader SHALL load and parse that file and produce a Merged_Config containing its values
6. IF a configuration file exists but is empty (zero bytes or contains only whitespace), THEN THE Config_Loader SHALL treat it as an empty configuration (equivalent to an empty YAML document) and SHALL NOT raise an error

### Requirement 2: Enforce Precedence Chain

**User Story:** As a user, I want function parameters and environment variables to override config file values, so that I can temporarily change behavior without editing my config files.

#### Acceptance Criteria

1. WHEN a function parameter is passed with a value that is not None, THE Bedrock_Pricing_Script SHALL use the function parameter value regardless of config file or environment variable values
2. WHEN a function parameter is None or not passed, and a corresponding environment variable is present in the environment with a non-empty string value, THE Bedrock_Pricing_Script SHALL use the environment variable value
3. WHEN neither a function parameter nor an environment variable provides a value, THE Bedrock_Pricing_Script SHALL use the Project_Config value if the key exists and is not null, otherwise the User_Config value if the key exists and is not null, otherwise the hardcoded default
4. THE Config_Loader SHALL resolve each setting independently through the Precedence_Chain so that one setting's source does not affect another setting's resolution
5. IF an environment variable value cannot be converted to the expected type for its corresponding setting, THEN THE Config_Loader SHALL ignore that environment variable, emit a warning identifying the variable name and expected type, and continue resolution with the next source in the Precedence_Chain

### Requirement 3: Validate Configuration File Structure

**User Story:** As a user, I want clear error messages when my config file has invalid content, so that I can fix problems quickly.

#### Acceptance Criteria

1. WHEN a configuration file contains invalid YAML syntax, THE Config_Loader SHALL report an error message including the file path and line number of the syntax error, and SHALL fall back to the next source in the Precedence_Chain for all settings in that file
2. WHEN a configuration file contains an unrecognized top-level section, THE Config_Loader SHALL emit a warning identifying the unrecognized section name and continue loading recognized sections from the same file
3. WHEN a configuration value has an incorrect type (e.g., string where integer is expected), THE Config_Loader SHALL report the setting name, expected type, and actual value, SHALL use the hardcoded default for that setting, and SHALL continue validating remaining settings in the file
4. IF a configuration file cannot be read due to a permissions error, THEN THE Config_Loader SHALL report the file path and permission issue and fall back to the next source in the Precedence_Chain
5. WHEN a configuration file contains multiple validation errors (type mismatches or unrecognized sections), THE Config_Loader SHALL report all detected errors and warnings before continuing, rather than stopping at the first error
6. WHEN a configuration file is empty or contains only comments, THE Config_Loader SHALL treat it as a valid file with no settings and SHALL use defaults from the next source in the Precedence_Chain

### Requirement 4: Support Reports Configuration Section

**User Story:** As a user, I want to configure report output preferences in my config file, so that generated reports consistently use my preferred directory, format, and naming conventions.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize a `reports` section with the following keys: `output_dir` (string), `format` (string, one of: "markdown", "json", "csv"), `retention_days` (integer, 1 to 3650), `naming_template` (string, maximum 128 characters), `include_metadata` (boolean), `auto_cleanup` (boolean)
2. WHEN `reports.output_dir` is specified, THE Bedrock_Pricing_Script SHALL write report files to that directory
3. IF `reports.output_dir` is specified and the directory does not exist, THEN THE Bedrock_Pricing_Script SHALL create the directory before writing report files
4. IF `reports.output_dir` is specified and the directory cannot be created or is not writable, THEN THE Config_Loader SHALL report an error message indicating the path and the access issue
5. WHEN `reports.retention_days` is specified and `reports.auto_cleanup` is true, THE Bedrock_Pricing_Script SHALL remove report files in the output directory that are older than the specified number of days at the start of each report generation
6. WHEN `reports.naming_template` is specified, THE Bedrock_Pricing_Script SHALL use the template to generate report file names, supporting the placeholders `{model}`, `{volume}`, `{timestamp}`, `{region}`, and `{format}`
7. WHEN `reports.format` is specified, THE Bedrock_Pricing_Script SHALL generate report output in the configured format

### Requirement 5: Support Defaults Configuration Section

**User Story:** As a user, I want to set my preferred region, tier, and history mode in the config file, so that I do not have to specify them on every invocation.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize a `defaults` section with the following keys: `region`, `tier_preference`, `history_mode`
2. WHEN `defaults.region` is specified and no region is provided via function parameter or environment variable, THE Bedrock_Pricing_Script SHALL use the configured region as the `region_filter` for pricing queries
3. WHEN `defaults.tier_preference` is specified and the preferred tier is available for the queried model, THE Bedrock_Pricing_Script SHALL select that tier instead of the default lowest-cost tier with prompt caching support
4. IF `defaults.tier_preference` is specified and the preferred tier is not available for the queried model, THEN THE Bedrock_Pricing_Script SHALL fall back to the default tier selection logic and emit a warning indicating the configured tier was not found
5. WHEN `defaults.history_mode` is specified with a value of "full" or "condensed", THE Bedrock_Pricing_Script SHALL use the configured history mode for token calculations
6. IF `defaults.history_mode` contains a value other than "full" or "condensed", THEN THE Config_Loader SHALL report the invalid value and the list of valid options, and fall back to the hardcoded default of "full"

### Requirement 6: Support Agent Defaults Configuration Section

**User Story:** As a user, I want to set default agent session parameters in the config file, so that repeated cost calculations use my standard workload profile without re-specifying every parameter.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize an `agent_defaults` section with the following keys: `questions_per_session` (positive integer), `input_tokens` (positive integer), `output_tokens` (positive integer), `system_prompt_tokens` (positive integer), `tools_passed` (non-negative integer), `tool_spec_tokens` (positive integer), `tools_invoked` (non-negative integer), `tool_call_tokens` (positive integer), `tool_result_tokens` (positive integer)
2. WHEN a cost calculation function is called without explicit token parameters, THE Bedrock_Pricing_Script SHALL use values from `agent_defaults` as defaults for each parameter not explicitly provided
3. WHEN a cost calculation function is called with explicit token parameters, THE Bedrock_Pricing_Script SHALL use the explicit parameters and ignore `agent_defaults` for those specific parameters
4. WHEN the `agent_defaults` section is present but only contains a subset of keys, THE Config_Loader SHALL apply the provided keys as defaults and use hardcoded defaults for any absent keys
5. IF an `agent_defaults` value violates its minimum constraint (e.g., zero or negative where positive is required, or negative where non-negative is required), THEN THE Config_Loader SHALL report an error message indicating the key name and the minimum allowed value, and SHALL discard that key's value in favor of the hardcoded default

2. WHEN `calculate_agentcore_cost()` is called without explicit infrastructure parameters, THE Bedrock_Pricing_Script SHALL use values from `agentcore_defaults` as defaults for each parameter not explicitly provided
3. WHEN `calculate_agentcore_cost()` is called with explicit infrastructure parameters, THE Bedrock_Pricing_Script SHALL use the explicit parameters and ignore `agentcore_defaults` for those specific parameters
4. WHEN the `agentcore_defaults` section is present but only contains a subset of keys, THE Config_Loader SHALL apply the provided keys as defaults and use hardcoded defaults for any absent keys
5. IF `agentcore_defaults.io_wait_pct` or `agentcore_defaults.eval_sampling_rate` is outside the range 0.0 to 1.0, THEN THE Config_Loader SHALL report the invalid value and fall back to the hardcoded default for that key

### Requirement 8: Support RAG Sub-Agent Defaults Configuration Section

3. WHEN `calculate_rag_subagent_tokens()` is called with explicit parameters in `token_params`, THE Bedrock_Pricing_Script SHALL use the explicit parameters and ignore `rag_defaults` for those specific parameters
4. WHEN the `rag_defaults` section is present but only contains a subset of keys, THE Config_Loader SHALL apply the provided keys as defaults and use hardcoded defaults for any absent keys

### Requirement 10: Support Business Value Defaults Configuration Section

#### Acceptance Criteria

THE Config_Loader SHALL recognize a `business_value_defaults` section with the following keys: `time_without_ai_min` (positive number, default: 20), `time_with_ai_min` (positive number, default: 10), `human_cost_per_hour` (positive number, default: 75), `revenue_per_hour` (positive number, default: 300), `agent_effectiveness_pct` (float 0.0–1.0, default: 0.65), `efficiency_factor_pct` (float 0.0–1.0, default: 0.60), `churn_without_ai_pct` (positive number, default: 2.0), `churn_with_ai_pct` (positive number, default: 1.0), `sales_increase_pct` (positive number, default: 10.0)

3. WHEN `calculate_business_value()` is called with explicit business value parameters, THE Bedrock_Pricing_Script SHALL use the explicit parameters and ignore `business_value_defaults` for those specific parameters

### Requirement 11: Support Capacity Configuration Section

**User Story:** As a user, I want to configure capacity planning assumptions in the config file, so that capacity checks use my validated traffic profile without re-entering values.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize a `capacity` section with the following keys: `peak_to_avg_ratio` (float, range 1.0 to 100.0, default: 3.0), `active_hours_per_day` (integer, range 1 to 24, default: 12), `active_days_per_month` (integer, range 1 to 31, default: 22), `max_tokens_setting` (integer, range 1 to 65536, default: 4096)
2. WHEN `check_capacity_fit()` is called without explicit capacity parameters, THE Bedrock_Pricing_Script SHALL use values from the `capacity` section as defaults for the corresponding function parameters (`peak_to_avg_ratio`, `active_hours_per_day`, `active_days_per_month`, `max_tokens_setting`)
3. WHEN `check_capacity_fit()` is called with explicit capacity parameters, THE Bedrock_Pricing_Script SHALL use the explicit parameters and ignore the `capacity` section values for those specific parameters
4. WHEN only a subset of `capacity` keys is specified in the configuration, THE Bedrock_Pricing_Script SHALL use the configured values for present keys and fall back to hardcoded defaults for absent keys (peak_to_avg_ratio: 3.0, active_hours_per_day: 12, active_days_per_month: 22, max_tokens_setting: 4096)
5. IF a `capacity` key value is outside its valid range or is not the expected numeric type, THEN THE Config_Loader SHALL report the key name, expected range, and actual value, and fall back to the hardcoded default for that key

### Requirement 12: Support Pricing Cache Configuration Section

**User Story:** As a user, I want to configure where pricing data cache files are stored and how long they remain valid, so that I can control disk usage and data freshness for the local pricing JSON files.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize a `pricing_cache` section with the following keys: `dir` (string, default: "~/bedrock_cache"), `max_age_days` (integer, range 1 to 365, default: 7), `auto_refresh` (boolean, default: false)
2. WHEN `pricing_cache.dir` is specified, THE Bedrock_Pricing_Script SHALL use the configured directory for reading and writing pricing data cache files (bedrock_pricing.json, bedrock_pricing_3p.json, bedrock_pricing_service.json, bedrock_pricing_agentcore.json, bedrock_quotas.json) instead of the default `~/bedrock_cache/`
3. IF `pricing_cache.dir` specifies a directory that does not exist, THEN THE Bedrock_Pricing_Script SHALL create the directory (including parent directories) before writing pricing cache files
4. IF `pricing_cache.dir` specifies a path that cannot be created or is not writable, THEN THE Config_Loader SHALL report an error message indicating the path and the access issue and fall back to the default `~/bedrock_cache/` directory
5. WHEN `pricing_cache.max_age_days` is specified, THE Bedrock_Pricing_Script SHALL consider pricing cache files older than the configured number of days as stale (replacing the hardcoded 7-day threshold)
6. IF `pricing_cache.max_age_days` is set to a value outside the range of 1 to 365, THEN THE Config_Loader SHALL report the invalid value and use the hardcoded default of 7 days
7. WHILE `pricing_cache.auto_refresh` is true, WHEN the Bedrock_Pricing_Script detects that pricing cache files are stale, THE Bedrock_Pricing_Script SHALL attempt to refresh the pricing cache before proceeding with queries
8. IF `pricing_cache.auto_refresh` is true and the pricing cache refresh attempt fails, THEN THE Bedrock_Pricing_Script SHALL proceed using the stale pricing cache files and emit a warning indicating the refresh failure reason

### Requirement 13: Support Behavior Configuration Section

**User Story:** As a user, I want to configure behavioral preferences like skipping confirmations and automatic capacity checks, so that the script adapts to my workflow without manual intervention each time.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize a `behavior` section with the following keys and types: `skip_confirmation` (boolean, default: false), `auto_capacity_check` (boolean, default: false), `detail_level` (string accepting "summary" or "full", default: "summary")
2. WHEN `behavior.skip_confirmation` is true, THE Bedrock_Pricing_Script SHALL skip interactive confirmation prompts before calculations
3. WHEN `behavior.auto_capacity_check` is true, THE Bedrock_Pricing_Script SHALL automatically run a capacity fit check after cost calculations using values from the `capacity` configuration section
4. WHEN `behavior.detail_level` is specified with a valid value ("summary" or "full"), THE Bedrock_Pricing_Script SHALL use the configured detail level as the default for output
5. IF `behavior.detail_level` is specified with a value other than "summary" or "full", THEN THE Config_Loader SHALL report the invalid value and the list of valid options, and fall back to the default value of "summary"
6. IF `behavior.auto_capacity_check` is true and required capacity parameters are not available in the `capacity` section or via other Precedence_Chain sources, THEN THE Bedrock_Pricing_Script SHALL skip the automatic capacity check and emit a warning indicating which capacity parameters are missing

### Requirement 14: Generate Configuration Template

**User Story:** As a user, I want to generate a commented config template file, so that I can see all available settings and their defaults without reading documentation.

#### Acceptance Criteria

1. WHEN the `--init-config` flag is passed, THE Bedrock_Pricing_Script SHALL generate a Config_Template file at `~/.bedrock_skills/config.yaml`
2. WHEN the target directory `~/.bedrock_skills/` does not exist, THE Bedrock_Pricing_Script SHALL create it before writing the Config_Template
3. THE Config_Template SHALL include YAML comments documenting each setting, its type, default value, and valid options
4. THE Config_Template SHALL include all eleven configuration sections: `reports`, `defaults`, `agent_defaults`, `rag_defaults`, `research_defaults`, `agentcore_defaults`, `business_value_defaults`, `capacity`, `pricing_cache`, `behavior`, `model_preferences` with all keys defined in Requirements 4 through 11 and 14 represented as commented-out entries set to their hardcoded default values
5. IF a config file already exists at the target path, THEN THE Bedrock_Pricing_Script SHALL prompt the user for confirmation before overwriting (if stdin is a TTY), and IF the user declines, THEN THE Bedrock_Pricing_Script SHALL abort the operation without modifying the existing file and print a message indicating no changes were made
6. WHEN `--init-config --force` is passed, THE Bedrock_Pricing_Script SHALL overwrite an existing config file without prompting for confirmation
7. WHEN `--init-config` is run in a non-TTY environment (stdin is not interactive) and a config file already exists, THE Bedrock_Pricing_Script SHALL abort without overwriting and print a message suggesting the use of `--force`
8. WHEN `--init-config` completes successfully, THE Bedrock_Pricing_Script SHALL print the absolute file path and a list of the eleven section names written to the Config_Template
9. IF the target directory cannot be created or the Config_Template file cannot be written due to a permissions or filesystem error, THEN THE Bedrock_Pricing_Script SHALL report an error message indicating the file path and the nature of the failure without modifying any existing files

### Requirement 15: Deduplicate Defaults Across SKILL.md Files

**User Story:** As a maintainer, I want default values defined in exactly one place (the Python function signatures), so that SKILL.md files don't drift out of sync when defaults are changed or overridden via config.

#### Acceptance Criteria

1. EACH SKILL.md file (bedrock-pricing, bedrock-capacity, agentcore-pricing, agent-business-value, bedrock-tier-advisor) SHALL remove inline default value columns from their "Parameter Defaults" or "Defaults" tables
2. EACH SKILL.md file SHALL retain parameter names, descriptions, and notes columns — only the hardcoded default value column is removed
3. EACH SKILL.md file SHALL include a "Configuration" section that states: defaults are defined in the Python function signatures, can be overridden via `~/.bedrock_skills/config.yaml`, and the user can run `--init-config` to generate a commented template showing all available settings with their current defaults
4. EACH SKILL.md "Configuration" section SHALL include the explicit statement: "Config values are defaults only. If the user specifies a value in their prompt, always use the user's value. Config defaults apply only to parameters the user has not mentioned."
5. THE Config_Template generated by `--init-config` SHALL serve as the single canonical reference for all overridable default values, with each entry showing the hardcoded default from the corresponding Python function signature
6. WHEN a hardcoded default value is changed in `bedrock_pricing.py`, THE Config_Template output SHALL automatically reflect the new value without requiring manual updates to SKILL.md files or the config template generation code

### Requirement 16: Support Model Preferences Configuration Section

**User Story:** As a user, I want to set default model selections by agent role in the config file, so that cost calculations automatically use my preferred models without specifying them each time.

#### Acceptance Criteria

1. THE Config_Loader SHALL recognize a `model_preferences` section with the following keys: `router` (string, default: "Claude Opus"), `general` (string, default: "Claude Sonnet"), `rag` (string, default: "Claude Haiku"), `research` (string, default: "Nova Lite"), `version` (string, default: "latest")
2. WHEN a cost calculation requires a model and the user has not specified one, THE Bedrock_Pricing_Script SHALL use the model preference for the corresponding role (router, general, rag, or research) as the `model_filter` for `query_model_pricing()`
3. WHEN `model_preferences.version` is "latest", THE Bedrock_Pricing_Script SHALL select the most recent version of the matched model from the pricing cache
4. WHEN `model_preferences.version` is a specific version string (e.g., "4.6"), THE Bedrock_Pricing_Script SHALL filter results to match that version
5. IF a model preference resolves to zero matches in the pricing cache, THEN THE Bedrock_Pricing_Script SHALL emit a warning indicating the configured model was not found and ask the user to specify a model
6. WHEN the user explicitly specifies a model in their prompt, THE Bedrock_Pricing_Script SHALL use the user-specified model regardless of `model_preferences` configuration

### Requirement 17: PyYAML Dependency

**User Story:** As a user, I want clear guidance on the PyYAML dependency, so that I can install it if needed and understand why it's required.

#### Acceptance Criteria

1. THE Bedrock_Pricing_Script SHALL require PyYAML (`import yaml`) for configuration file parsing
2. IF PyYAML is not installed, THE Config_Loader SHALL emit a clear error message stating "PyYAML is required for config file support. Install with: pip install pyyaml" and SHALL fall back to hardcoded defaults for all settings
3. THE `--init-config` generated template SHALL NOT require PyYAML to be written (template generation uses string formatting only)
4. THE Bedrock_Pricing_Script SHALL NOT include a custom YAML parser — PyYAML is the sole parsing mechanism
