# Requirements: File-Based Report Output

## Overview

`calculate_agent_session_compounded_cost()` currently returns a large result dict (~5-10K tokens when serialized as full detail). This feature changes the function to always write the full detailed markdown report to a file and return only a compact summary containing key metrics, the file path, and the `capacity_profile` for downstream use. The `detail_level` parameter is removed entirely.

## User Stories

### US-1: Agent gets lean context
As an AI agent skill, I want the cost function to always return a compact summary so that my context window stays lean instead of carrying the full 5-10K token result dict.

### US-2: User gets persistent report
As a user, I want every cost estimate automatically saved as a timestamped markdown file so that I can compare estimates, share them, and reference them later.

### US-3: Agent can still do capacity planning
As an AI agent skill, I want `capacity_profile` included in the summary so that I can pass it directly to `check_capacity_fit()` without reading the file back.

## Requirements

### 1. Single Return Behavior (no detail_level)
- 1.1. Remove the `detail_level` parameter from `calculate_agent_session_compounded_cost()`
- 1.2. The function always computes the full detailed result internally
- 1.3. The function always writes the full markdown report to a file
- 1.4. The function always returns a compact summary dict containing key metrics, file_path, and capacity_profile
- 1.5. If file write fails (permissions, disk full), fall back to returning the full result dict inline with `_file_write_failed: true` and a clear stderr warning explaining the error, the impact (increased latency and token usage), and how to fix it (specify a writable folder path)

### 2. Compact Summary (returned to agent)
- 2.1. Summary dict contains: `file_path`, `sessions_per_month`, `monthly_total`, `annual_total`, `session_total`, `session_total_no_cache`, `savings_pct`, `main_agent_session_cost`, `subagent_session_cost`, `recommended_ttl`, `top_cost_driver`, `capacity_profile`
- 2.2. `capacity_profile` is the full dict (same as today) — enables direct handoff to `check_capacity_fit()`
- 2.3. Numeric values are rounded: monthly/annual to 2 decimals, session to 6 decimals, savings_pct to 1 decimal
- 2.4. Summary is significantly smaller than the full result dict — the exact token count depends on the number of sub-agents in capacity_profile

### 3. Report File Location
- 3.1. Default output directory: `~/bedrock_reports/`
- 3.2. Directory is created automatically if it doesn't exist
- 3.3. Configurable via `reports.output_dir` in the YAML config system
- 3.4. Overridable via explicit `output_path` parameter (full path to file)
- 3.5. If `output_path` is provided, it takes precedence over config and default

### 4. Report File Naming
- 4.1. Default naming pattern: `{model}_{volume}_{timestamp}.md`
- 4.2. `{model}` — sanitized model name (lowercase, spaces→hyphens, e.g., "claude-sonnet-4.6")
- 4.3. `{volume}` — sessions per month with suffix (e.g., "10k-sessions", "1m-sessions")
- 4.4. `{timestamp}` — `YYYYMMDD-HHMMSS-XXXX` format (local time + 4-char random hex to prevent collisions)
- 4.5. Configurable via `reports.naming_template` in config
- 4.6. Example: `claude-sonnet-4.6_10k-sessions_20260526-143022-a7f3.md`

### 5. Report File Content
- 5.1. Report is a complete markdown document generated by `_format_full_output()`
- 5.2. Report includes YAML front-matter metadata block at the top (if `reports.include_metadata` is True)
- 5.3. Front-matter contains: generated_at (ISO 8601), model, region, sessions_per_month, session_total, monthly_total, annual_total, savings_pct, inputs_hash (SHA-256 of input parameters, first 16 chars)
- 5.4. Report body follows the standard section order defined in `_format_full_output()`
- 5.5. If `_format_full_output()` cannot produce output (missing optional params like cache_status), write a simplified report with token_table + cost summary

### 6. Versioning and Comparison
- 6.1. Each run creates a new file (timestamp-based versioning)
- 6.2. Tweaked inputs = new file (enables side-by-side comparison)
- 6.3. No explicit V1/V2 numbering — timestamps provide ordering
- 6.4. `inputs_hash` in front-matter is an audit/metadata field for traceability

### 7. Retention and Cleanup
- 7.1. Default retention: 30 days (configurable via `reports.retention_days`)
- 7.2. `--cleanup-reports` CLI flag deletes reports older than retention_days
- 7.3. Cleanup is manual only unless `reports.auto_cleanup` is True in config
- 7.4. Cleanup prints count of deleted files and freed space
- 7.5. Cleanup only deletes files matching the report naming pattern (derived from `reports.naming_template`), not arbitrary `.md` files in the directory
- 7.6. The pattern matcher must adapt if `reports.naming_template` changes — derive the match regex from the template at cleanup time

### 8. Error Handling
- 8.1. If output directory cannot be created or file write fails, try the default directory (`~/bedrock_reports/`) as fallback. If that also fails, fall back to returning the full result dict inline
- 8.2. Emit a clear warning to stderr: "⚠️ Report: Failed to write report to '{path}'. Returning full result inline. This increases token usage and latency. Specify a writable folder via reports.output_dir in ~/.bedrock_skills/config.yaml or pass output_path='/writable/path/report.md'."
- 8.3. The inline fallback result includes a `_file_write_failed: true` key so the agent can detect the fallback
- 8.4. Never raise an exception for file I/O failures — always return a result (either summary or inline fallback)

### 9. Config Integration
- 9.1. `reports.output_dir` controls default output directory
- 9.2. `reports.naming_template` controls filename pattern
- 9.3. `reports.include_metadata` controls whether YAML front-matter is included
- 9.4. `reports.retention_days` controls cleanup threshold
- 9.5. `reports.auto_cleanup` controls whether old reports are deleted on each write
- 9.6. `reports.format` is reserved for future use (json/csv) — only markdown is generated in this PR. The config template comment should note this.
