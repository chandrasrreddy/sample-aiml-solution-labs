## Overview

An independent QA judge that verifies Bedrock agent cost and business value estimates. Implements every formula from `pricing_spec_v1.1.md` to independently recalculate all values, then compares against the estimate under test. Produces a detailed pass/fail report with per-field verification and highlights any discrepancies.

The judge follows a strict protocol: self-test first (reproduce spec reference examples), then evaluate the target estimate.

## Workflow

### Step 1: Self-Test (Mandatory)
- **Mode**: `deterministic`
- **Tool**: `run_python`
- **Input**: None (uses hardcoded reference examples from the spec)
- **Output**: Self-test results for all 4 reference examples (A, B, C, D)

Load the bundled QA judge script and run the self-test:

```python
exec(open(os.path.expanduser("~/.quickwork/skills/bedrock-cost-qa/scripts/qa_judge.py")).read())
self_test_results = run_self_test()
```

This reproduces Examples A, B, C, D from the spec and checks every expected value. The self-test ensures the judge's own formulas are correct before it evaluates external estimates.

- **Validate**: ALL 4 examples pass (every field matches expected values within 0.01% tolerance)
- **On failure**: **ABORT** — do NOT proceed. The judge's logic is broken. Report which example failed and which field diverged.

### Step 2: Load Inputs
- **Mode**: `deterministic`
- **Tool**: `file_read`
- **Input**: `{{estimate_file}}`, `{{test_spec_file}}`
- **Output**: Parsed estimate and test spec (or just estimate if no spec provided)

Read the estimate file. If it's JSON, parse directly. If it's Markdown, extract the key values (model cost, AC cost, business value, etc.) using pattern matching.

If `{{test_spec_file}}` is provided, load the expected values from it. Otherwise, extract the input parameters from the estimate and compute expected values on-the-fly (Step 3).

- **Validate**: Estimate file exists and contains recognizable cost data
- **On failure**: Report what's missing; ask user for the correct file path

### Step 3: Independent Recalculation
- **Mode**: `deterministic`
- **Tool**: `run_python`
- **Input**: Input parameters extracted from estimate or test spec
- **Output**: Complete set of independently computed expected values

```python
judge_results = judge_estimate(
    inputs=estimate_inputs,    # from the estimate file
    test_spec=test_spec_data   # from test spec file, or None
)
```

The judge computes every intermediate value from scratch using the spec formulas:
1. Token math (base_context through monthly totals)
2. Cache splits (Q1, Q2, session, monthly)
3. All cost line items
4. Business value (if included in the estimate)

- **Validate**: All verification identities hold for the judge's own computation
- **On failure**: Report internal consistency error

### Step 4: Compare & Verdict
- **Mode**: `deterministic`
- **Tool**: `run_python`
- **Input**: Judge's expected values vs. estimate's actual values
- **Output**: Per-field comparison with pass/fail/warn status

For each field, compute:
```
pct_diff = abs(expected - actual) / expected × 100
```

Verdict per field:
- ✅ **PASS**: pct_diff ≤ 0.1%
- ⚠️ **WARN**: 0.1% < pct_diff ≤ 1.0% (likely rounding or minor model difference)
- ❌ **FAIL**: pct_diff > 1.0%

Overall verdict:
- ✅ **PASS**: All fields pass
- ⚠️ **WARN**: Some fields warn but none fail (common for AgentCore prices which vary)
- ❌ **FAIL**: Any field fails

- **Validate**: Verdict is consistent (no contradictions between field-level and overall)
- **On failure**: Report the most egregious failures first

## Lessons Learned

### Do
- ALWAYS run the self-test first — it catches formula bugs before they produce false verdicts
- Report ALL intermediate values, not just final costs — this makes debugging failures trivial
- Use relative tolerance (%) not absolute ($) for comparisons — a $0.50 difference matters at $100 but not at $100,000

### Don't
- Don't skip the self-test even if it was run recently — the script state might have changed
- Don't use the bundled `calculate_agent_cost_with_incremental_caching()` function from the pricing skill — it has a known minor divergence on Q1 Turn 0 cache classification. The judge implements the spec directly.
- Don't treat AgentCore price differences as failures — AC prices vary by region and may not exactly match the estimate if it used different rate assumptions. Use WARN level for AC discrepancies.

### Common Failures
- **Q1 Turn 0 cache classification**: The spec says Turn 0 of Q1 is entirely cache_write (base_prompt). Some implementations treat T_user + T_rag as regular input. Both are valid; the judge uses the spec's approach but warns (not fails) on this specific discrepancy.
- **Rounding**: Token counts are integers; costs may differ by pennies due to rounding. Use 0.1% tolerance.
- **AgentCore pricing**: AC unit prices change. If the estimate used different AC rates, field-level differences are expected — check internal consistency (components sum to total) instead.

### When to Ask the User
- When the estimate file format is unrecognizable
- When the estimate doesn't include enough input parameters to reconstruct expected values (and no test spec is provided)
